The tech stack

Mozaiq is a TypeScript monorepo (npm workspaces) with three consumer surfaces — the mobile app, an internal web evaluation tool, and a CLI pipeline — all sharing a single core package so the solver that ran during generation is byte-for-byte the same one running on your phone.

The puzzle-generation algorithm

Three base colors — red, yellow, blue — combine into three mixed colors wherever two sections cross: yellow + blue = green, red + yellow = orange, red + blue = purple. These mixing rules are the constraints.

Solver

Exhaustive backtracking with constraint propagation. Given a board mask and a set of hint cells, the solver enumerates legal section-color assignments, pruning branches the moment a mixed-color cell contradicts its adjacent sections. Configurable timeout (default 30 seconds, validation runs go up to 3 minutes) and a solution cap (default 1,000; uniqueness audits push to 5,000). Lives in src/lib/solver.ts and runs in a Web Worker on the desktop tool so the UI thread never blocks.

Generator

Candidate layouts are pulled from external crossword corpora (NYT, WordsUp, iPuz) via importers in src/lib/layoutBuild/, ranked by viability and propagation quality, then handed to the puzzle generator which places hint cells and runs the solver to check uniqueness. Anything that doesn't produce a single, no-guess solution is rejected and regenerated. A recent commit literally reads "Replace puzzle bundle with quality-enforced regeneration" — the 8,084-puzzle shipping bundle is the output of running this loop until everything passes.

Difficulty rating

scripts/analyze-puzzle-difficulty.ts labels every generated puzzle Easy / Medium / Hard / Expert from propagation depth, hint density, and search-space metrics — not from board size. A small 7×7 with heavy ambiguity can rate Expert; a sparse 15×15 with strong propagation can rate Easy.

The content pipeline

41 TypeScript CLI scripts plus 2 shell scripts orchestrate the journey from raw external crossword grids to a curated, validated, difficulty-labeled puzzle bundle ready to ship inside the app. Each stage is one npm script.

1 · Ingest

Pull external corpora: download-wordsup-grids, restructure-nyt-crosswords, import-masks, iPuz importer.

2 · Extract layouts

make-layouts extract finds valid Mozaiq-compatible layouts (no illegal 2×2 blocks, valid section topology) inside the corpus.

3 · Rank

rank-mask-library, find-golden-templates, rank-hard-layouts — score layouts by viability and propagation quality.

4 · Generate puzzles

batch-make-puzzles assigns section colors, places hints, and runs create-hints for each chosen layout.

5 · Validate uniqueness

validate-solution-distinctness, audit-solution-counts, validate-bulk-solver — re-solve every puzzle and confirm exactly one solution, with no guess-required cells.

6 · Difficulty rate

analyze-puzzle-difficulty labels each puzzle by propagation metrics, then curate-hard-puzzles picks the top expert set.

7 · Bundle & ship

merge-label-puzzles assembles the final JSON, dropped into the app and shipped via the iOS build script.

Agent-assisted workflow

Mozaiq was built with AI coding assistants from day one, but the interesting part isn't that — it's the discipline around it. Two patterns kept the code coherent at scale:

Phased prompts in prototype_prompts/

The repo preserves the original six-phase prompt set that scaffolded the codebase, each phase assuming the prior commit:

  1. Product Spec — game rules, color mixing, section model
  2. Project Setup — Vite / React / TypeScript scaffold
  3. Board & Sections — parser, section detection, topology
  4. Game Logic & UI — game state, color assignment, interaction
  5. Puzzle Generator & Solver — algorithm, generator, hint placement
  6. Polish & QA — tests, edge cases, performance

Phasing prevents the agent from drifting on the hard modules (solver, section detection) — each phase has a narrow scope and a clear acceptance target.

Autonomous validation loops

scripts/run-solver-validation-loop.ts behaves like an automated QA agent: it runs the solver against the bulk puzzle set in rounds, analyzes the report, and either re-runs or exits non-zero. It's the kind of harness that lets you make a generator change at midnight and wake up to a clean report — or a clear list of what regressed.

Quality-enforced regeneration

The puzzle bundle isn't generated once and trusted. It's regenerated until it passes every audit — uniqueness, no-guess solvability, all-three-color hint coverage. The constraint that every puzzle must use all three base colors as hints (commit "Enforce all-3-color hints and raise retry floor") lives in the pipeline itself, not as a convention the generator is trusted to follow.

Testing approach

Test coverage is deliberately concentrated on the parts where regressions silently corrupt content — not on UI surface area, which is easier to eyeball.

  • Unit tests (Vitest) on solver correctness, layout extraction, layout sampling, layout ordering, and the typed serialization contracts between pipeline stages.
  • Reference fixtures in fixtures/: canonical layouts, canonical masks, and both known-valid and known-invalid completed boards — so the solver and validator have ground truth to regression-test against.
  • End-to-end validators in scripts/ act as integration tests across the whole pipeline and produce machine-readable JSON reports (bulk-solver-report.json, assignment-count reports, distinctness reports).
  • Meta-tests like validate-solver-time-vs-solutions.ts assert that more solver time yields more solutions — a sanity check on the timeout knob itself.

Release engineering

Release coordination is scripted and intentional, not ad-hoc.

  • Two environments — separate dev and prod Supabase projects, with .env and .env.production in the mobile app.
  • deploy-supabase.sh promotes migrations dev → prod behind a typed-yes confirmation gate so you can't fat-finger a prod push.
  • build-and-submit-ios.sh runs the full local macOS pipeline: archive → IPA export → TestFlight submission, with the order documented (backend live → app build → TestFlight) so a breaking schema change can't ship ahead of the client that handles it.
  • Account self-deletion is wired end-to-end — Postgres migration, RLS policy, mobile UI, all the way to the "Delete my account" button in the Account sheet.

What this proves

Mozaiq isn't just a puzzle game — it's a system. A constraint solver that's the same code on the phone and in the validator. A generator that earns its output by passing audits. A release pipeline that knows what order to ship things in. An agent-assisted workflow that produces a real, testable, shippable product instead of a demo.

Built solo. Documented as it was built. Ready to scale.
